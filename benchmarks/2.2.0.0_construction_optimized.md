### Changes (vs 2.2.0.0_construction.md baseline)

- **Lazy FastEncoder:** Deferred FastEncoder (`FrozenDictionary<string, int>`) initialization from construction to first Encode/CountTokens call
- **Quadratic probing:** Replaced linear probing with triangular number probing in TokenEncoder hash table

### Results

| Metric | Before | After | Improvement |
|--------|--------|-------|:-----------:|
| **o200k construction** | 31.56 ms | 12.39 ms | **2.5x faster** |
| **cl100k construction** | 14.08 ms | 2.56 ms | **5.5x faster** |
| **o200k memory** | 43.93 MB | 13.12 MB | **70% reduction** |
| **cl100k memory** | 21.47 MB | 6.48 MB | **70% reduction** |
| **FirstCall (construction + encode)** | ~31 ms | ~31 ms | Same (deferred cost) |

### Construction Phase Profiling

| Phase | Time | % of Total | Memory |
|-------|------|-----------|--------|
| Binary resource loading (`LoadEncodingFromManifestResource`) | 10.3 ms | **83%** | 11.1 MB |
| TokenEncoder hash table build | ~2 ms | 16% | 2.0 MB |
| Regex compilation | 0.024 ms | 0.2% | 0.05 MB |
| Special tokens + other | <0.1 ms | <1% | <0.1 MB |

### Key Findings

- **Binary resource loading is the bottleneck** (83% of construction time)
- FastEncoder memory (~30MB for o200k) is deferred to first use, dramatically reducing construction allocation
- Regex compilation is trivial (24μs) — no benefit from parallelizing it
- Robin Hood hashing was tested but **regressed** (14.1ms vs 12.4ms) — at 38% effective load factor, quadratic probing has near-zero collisions, making Robin Hood's swap chains pure overhead
- Further construction optimization requires reducing binary resource loading cost (memory-mapped IO, pre-built hash table in binary format, etc.)
